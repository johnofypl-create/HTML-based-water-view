# 物理水（Physics-based Water）技术调研报告

> 调研目标：为项目接入「地形改造」功能做前置技术选型。核心需求——当用户把地形高度降低到水平面以下时，水要能**实时流入低洼区域并完成填充**。
> 调研日期：2026-07-09 ｜ 适用项目：coast-diorama（Three 0.175 / R3F 9 / drei 10 / Vite 8 / WebGL2）

---

## 0. 项目现状与需求约束（选型前提）

先看清楚我们脚下有什么，避免"为了物理水而推倒重来"。

**当前水实现（`src/water/`）**
- 一个**全局平面**：`PlaneGeometry(WORLD_SIZE*1.6, ..., 160, 160)` 放在常量 `WATER_LEVEL` 上，挂自定义 `waterMaterial`（Gerstner 波 + 深度着色）。
- 本质是一张"无限海平面"，**没有任何按格子填充低洼的逻辑**——只要地形低于水平面，平面会直接"盖"在上面，不会出现"水从海边流进来灌满洼地"的过程。

**当前地形实现（`src/world/terrainGeometry.ts`、`src/utils/terrain.ts`）**
- 高度来自解析函数 `heightAt(x, z)`，已有一套**高度场烘焙基础设施**：`getHeightFieldTexture()` 把 `heightAt` 采样成一张 `HEIGHT_TEX_SIZE × HEIGHT_TEX_SIZE` 的 `DataTexture`（RedFormat / FloatType），水着色器已经在采样它做深度色。
- 地形几何是网格 + 裙边（截面岩层感）。后续"地形改造"只需让 `heightAt` 可写、并增量更新这张高度场纹理即可。

**关键结论（对选型影响极大）**
1. 我们**已经有一张地形高度场纹理**——这正是高度场/浅水方案最想要的"地形输入"，零额外成本就能复用。
2. 当前是 **WebGL2（WebGPURenderer 未启用）**。GPGPU 在 WebGL2 下用"渲染到浮点纹理 + ping-pong"就能做，**不必为了物理水而迁移渲染器**（迁移是另一个量级的风险）。
3. 需求是"灌满洼地"，不是"飞溅的水花 / 流体特效"。这把粒子系那一类的优先级直接拉低。

## 1. 主流方案分类

按"水的表示方式"分成几类，对应你点名的三个方向：

| 类别 | 水的表示 | 代表方法 | 典型用途 |
|---|---|---|---|
| **A. 粒子系统流体** | 离散粒子 + 邻域相互作用 | SPH / PBF / MPM | 飞溅、入水、小规模交互 |
| **B. 高度场 / 浅水方程（网格水面）** | 每个格点一个"水深"标量 | 浅水方程 SWE / Virtual Pipes（虚拟管道法） | **洪水淹没、洼地填充、河流** |
| **C. 网格 Eulerian 流体（体素/不可压缩）** | 速度场 + 压力场（全 3D 或 2.5D） | Stable Fluids / 有限体积 SWE | 烟雾、大尺度 CFD、精确水动力学 |
| D. 屏幕空间近似 | 深度图 + 后处理 | Screen-space fluid / 深度融合 | 水面视觉润色（**非填充逻辑**） |
| E. 预计算/隐式面 | Level-set / Metaball / Marching Cubes | 等值面 | 动态地形下"画"出水面（**非实时流体**） |
| F. 混合（推荐落点） | B 做填充 + 现有 Gerstner/FFT 做表面细节 | — | **我们最该走的路** |

> 你列的三类正好对应 A / B / C。D、E 不是"物理水逻辑"，而是"怎么把水画好看"或"怎么糊弄出水面"，本调研把它们作为补充项，不纳入主推。

## 2. 三方案横向对比（性能 / 真实度 / 地形交互）

> 性能数字为**量级估计**（基于公开 demo 与通用经验），不是本项目实测；落地前需按真实网格尺寸打样验证。

### A. 粒子系统（SPH / PBF / MPM）

- **原理**：把水当成 N 个粒子，靠邻域密度/压力求加速度。要填满一片洼地，就得"注入"足够多粒子。
- **实时性能**：GPU 粒子（WebGPU Compute / WebGL transform feedback）可到 10万~百万级，但**每帧邻域搜索是瓶颈**（网格哈希 O(N)，常数大）。CPU 朴素 O(N²) 仅能撑 1~2k 粒子。
- **真实度**：飞溅、水花、入水**最真实**，但"平静灌满洼地"这种需求反而体现不出优势。
- **与动态地形交互**：**这是最大痛点**。每个粒子都要和地形做碰撞检测——通常要采样地形高度场 / SDF，洼地的"侧壁、底部"都得算。地形一改，碰撞场要实时重建；洼地越深、壁越陡，粒子越容易穿透或卡住。
- **填充低洼的代价**：要让洼地被"填满到某个水位"，本质是约束大量粒子停在某个等势面——粒子法对此既不经济也不稳（粒子会抖动、需要压力松弛很多步才平）。
- **适用**：角色入水、船尾浪、小水坑交互。**不适合"地形改造→大片淹没"**。

### B. 高度场 / 浅水方程（网格水面）★

- **原理**：把世界切成 `G×G` 网格，每格只存一个标量 `h`（该格水深）。用浅水方程或它的简化版（Virtual Pipes / Pipe Model）在相邻格之间算"流量"，水自然从高总水头（地形高 + 水深）流向低总水头。
- **实时性能**：每步每格只和 4 个邻居算一次，**O(G²)** 且与粒子数无关。GPU ping-pong：256²（6.5万格）每帧 <1ms；512² 轻松；1024²（百万格）也仅几 ms。CPU 跑 256² 也够用（几 ms）。**这是扩展性最好的一类**。
- **真实度**：能正确表现**水位上涨、洼地灌满、连通流域、漫过垭口、回流**——正好命中我们的需求。表面"波浪感"靠在水位面上叠加现有 Gerstner/FFT 细节即可，不靠物理求解。
- **与动态地形交互**：**天然契合**。地形就是另一张高度场纹理 `T(x,z)`，水头 = `T + h`。地形一改（更新 `T` 纹理），下一帧流量计算自动把水"引"进新出现的洼地。**无需任何专门的碰撞检测**——采样高度场即碰撞。
- **稳定性**：Virtual Pipes 需用"管道容量钳制"（`flux = min(capacity, availableWater)`）避免一步把一格水抽干导致数值爆炸；"Extended Virtual Pipes"（2018）进一步解决了"完全淹没通道"的精度问题。
- **适用**：**洪水/洼地填充/河流/湖泊/可调地形水景**——就是我们的场景。

### C. 网格 Eulerian 流体（体素 / 不可压缩 SWE 有限体积）

- **原理**：在网格上解速度场 + 压力场（全 3D 或 2.5D 分层），比浅水方程更"完整"（能模拟垂直方向速度、回流涡旋）。
- **实时性能**：每步要解压力泊松方程（多重网格/共轭梯度），**比 B 贵一个数量级**。2D/2.5D 大城市洪水 GPU 方案（如 BG_Flood、Nature 2026 的城市内涝 GPU 模型）能做到实时，但那是 HPC 级实现，前端复刻成本高。
- **真实度**：最高，能出涡旋、分层流。但对"灌满一个洼地"而言，多出来的物理精度**基本用不上**。
- **与动态地形交互**：同 B，地形即边界条件，耦合没问题；但求解器对地形突变（玩家一铲子挖下去）的鲁棒性要靠很多边界处理兜底。
- **适用**：科研级水动力学、城市内涝仿真、需要强物理保真的场合。**对本项目属于过度工程**。

## 3. 水体与可变地形的碰撞检测 & 流量计算（核心机制）

### 3.1 为什么 B 类几乎"没有碰撞检测"问题

在高度场框架下，"水"和"地形"是**同一套网格上的两张标量场**：

```
地形高度场  T[i,j]   ← 来自 heightAt / 可编辑
水深场      h[i,j]   ← 物理求解得到
水面高度    S[i,j] = T[i,j] + h[i,j]
```

- **碰撞 = 采样**：水永远"贴着"地形走，不存在穿透问题。
- **地形实时编辑**：玩家改地形 → 更新 `T` 纹理 → 下一帧 `S = T + h` 自动反映新洼地，流量项自动把水推过去。
- **海平面作为水源**：把世界边界（或 `T < WATER_LEVEL` 的"海"侧）设为恒定水源边界（总水头恒为 `WATER_LEVEL`）。洼地一低于海平面，海侧就持续供水直到洼地水位 = 海平面（连通时）或达到局部平衡（被高地隔断时，形成内陆湖）。

### 3.2 Virtual Pipes（虚拟管道法）流量公式（可直接落地）

这是浅水方程在游戏/实时里最常用、最稳的离散化，比完整 SWE 好实现：

```
对每个格 (i,j)，与 4 个邻居各连一根"虚拟管道"：
  某邻居 k 的"总水头" dH = (T[i,j] + h[i,j]) - (T[k] + h[k])
  管道流量 f_k = clamp( A * dH / L ,  -h[i,j] ,  +h[k] )   // 受管道截面积、格距、可供水量的钳制
  若 dH>0 → 水流出本格；dH<0 → 水流入本格
更新：h[i,j] += dt * Σ(流入 - 流出) / cellArea
```

要点：
- `clamp` 上界用邻居可用水量、下界用本格水量，**保证水不会变负、也不会凭空产生**——这是数值稳定的命门。
- 常数 `A`、`L`、`dt` 决定了"流速感"，调参即可（想让水"涌"得快就加大 `A/dt`）。
- 边界：海侧格 `h` 锁定为 `WATER_LEVEL - T`（即水面恒在海平面）；世界外缘设为无流（墙）。

### 3.3 粒子法（A）在此处的碰撞代价对照

若坚持用粒子，洼地填充需要：① 把地形烘焙成**有向距离场 / 高度场**供粒子查询；② 每帧每粒子做"高度场采样 + 投影回地表 + 法向反弹"；③ 洼地侧壁陡 → 需细分碰撞 + 子步，否则穿透；④ 要让"水位"稳定，需大量压力松弛步，且粒子数 ∝ 洼地体积。综上：**实现难、稳不住、贵**。

## 4. 大规模地形的扩展性 & 性能开销

| 方案 | 计算复杂度 | G=256² 每帧 | G=1024² | 内存（G² 浮点） | 地形实时编辑成本 |
|---|---|---|---|---|---|
| A 粒子 | O(N) + 邻域搜索常数大 | 10k 粒子勉强；填满洼地需灌数万~十万粒子 | 难 | 粒子态 N×若干 | **高**（重建碰撞场 + 防穿透） |
| **B 高度场** | **O(G²)**，每格只算 4 邻居 | **<1ms（GPU）** | 几 ms | 256²≈0.5MB / 1024²≈8MB | **极低**（改一张纹理） |
| C Eulerian | O(G²·迭代) 解压力 | 数 ms~数十 ms | 重 | 多张场，数倍于 B | 中（边界条件维护） |

**扩展性结论**：高度场方案与"世界大小"呈单纯的平方关系、且无隐藏的邻域搜索爆炸，**是唯一能平滑扩展到项目级大世界（1024² 甚至更高）且不掉帧的路线**。粒子法在"填满一大片洼地"时粒子数随体积线性膨胀，最先撞墙。

> 落地建议：物理网格分辨率**不必等于**地形渲染分辨率。物理 `G` 取 256~512 即可表现淹没有效；渲染水面时在此基础上插值 + 叠加 Gerstner 细节，视觉精度由渲染网格决定，物理由 `G` 决定，二者解耦。

## 6. 推荐技术路线（针对"动态地形交互"核心需求）

### 结论先行

**采用「高度场浅水方程（Virtual Pipes 方法）+ GPU ping-pong 求解」，并在水面渲染上叠加现有的 Gerstner/FFT 表面细节。** 这是与项目现状耦合最紧、动态地形交互最自然、扩展性最好、实现难度适中的一条路。**不采用粒子法**（需求不匹配、碰撞与填充都贵），**不采用完整 Eulerian**（过度工程）。

### 为什么是它（一句话逻辑链）

> 我们已经有 `getHeightFieldTexture()`（地形高度场）→ 再加一张 `水深场 h` → 每帧用 Virtual Pipes 算流量 → `S = T + h` 就是水面 → 改地形只改 `T` 一张纹理，水自动灌进来。**零新增基础设施、零专门碰撞检测、天然支持地形实时编辑。**

### 落地架构（与现有代码对接）

```
                    ┌─────────────── 输入：地形高度场 T（已有，可编辑）───────────────┐
                    │  getHeightFieldTexture() / 地形改造后增量更新                       │
                    ▼
   [GPU ping-pong 求解器]  (WebGL2 下用 three 的 GPUComputationRenderer 或自写 FBO)
      输入：上一帧 h_tex + T_tex + WATER_LEVEL（海侧水源边界）
      每步：算 4 邻居 Virtual Pipes 流量 → 更新 h → 钳制非负
      输出：新 h_tex
                    │
                    ▼
   [水面渲染网格]  vertexY = T + h  （仅 h>ε 且 T<WATER_LEVEL 处显示水体）
      叠加现有 Gerstner 法线/波细节 → 复用 waterMaterial 的深度着色与焦散
                    │
                    ▼
   [现有后期]  Bloom / DoF / 色调映射（已就绪）
```

### 渲染策略（两种可选，推荐第一种）

1. **统一高度场水面（推荐）**：整个世界用一张 `G×G` 水深场。`h>ε` 的格才渲染水体，洼地被海水源填满后自然呈现"内陆湖/新海湾"。和现有"全局海平面平面"可平滑过渡——初始全场 `h = max(0, WATER_LEVEL - T)` 即等价于现在的海平面。
2. **混合（海用平面 + 内陆用高度场）**：开放海域仍用现有大平面保视觉，仅对 `T < WATER_LEVEL` 的"受淹区"跑高度场。更复杂、边界处理麻烦，除非性能吃紧否则不推荐。

### GPU 路线选择（重要取舍）

- **现在就做（推荐起步）**：**WebGL2 + 浮点渲染目标 ping-pong**，用 three 自带 `GPUComputationRenderer`（`examples/jsm/misc/GPUComputationRenderer.js`，WebGL2 下支持浮点 RT）。**不迁移渲染器**，风险最低，今天就能在 0.175 上跑。
- **后续可选**：若将来启用 `WebGPURenderer`，把求解器改写成 TSL Compute（`three.js` 官方已有 `webgpu_compute_texture_pingpong` 范例），代码更干净、带宽更高。但这是独立决策，不应绑死本次需求。

### 参考实现（可直接扒代码）

- **`lisyarus/webgpu-shallow-water`** — Virtual Pipes 方法的 WebGPU 参考实现，公式与钳制逻辑最清晰，可作为算法蓝本（即便我们先用 WebGL2，算法一致）。
- **`aeplay/WebFlood`** — 浏览器内半拉格朗日浅水洪水模拟，纯前端可跑，验证"淹没"表现。
- **three.js 官方 `webgpu_compute_texture_pingpong`** — 我们的 WebGPU 演进范本。
- **three 自带 `GPUComputationRenderer`** — 我们现在最该直接用的 WebGL2 GPGPU 工具。

### 分阶段实施建议（降低风险）

| 阶段 | 内容 | 里程碑 |
|---|---|---|
| P0 原型 | 在 WebGL2 下用 `GPUComputationRenderer` 跑通 64²/128² 的 Virtual Pipes，固定地形看"水往低处流" | 控制台能打印水位收敛 |
| P1 接入地形 | 接 `getHeightFieldTexture()` 作 `T`，海侧设 `WATER_LEVEL` 水源边界 | 初始状态 ≈ 现有海平面 |
| P2 动态编辑 | 地形改造时增量更新 `T` 纹理，观察水实时灌入新洼地 | 挖坑→水涌入→填满 |
| P3 渲染融合 | 水面网格 `Y=T+h`，叠加 Gerstner 细节 + 现有深度着色/焦散 | 视觉与现版本一致且多了淹没过程 |
| P4 调参与扩展 | 调 `A/L/dt` 流速感；上 256²/512²；必要时做物理-渲染分辨率解耦 | 大世界实时不卡 |

## 7. 风险与备注

- **数值稳定性**：务必实现流量钳制（`clamp` 到可用水量），否则地形突变时易爆炸。先 P0 小网格验证收敛再放大。
- **性能数字为量级估计**：本报告 G=256²/1024² 的耗时是通用经验量级，非本项目实测；P0 阶段务必用真实网格打样。
- **不迁移渲染器**：WebGL2 的 GPGPU 足以支撑本需求，避免因切换 WebGPU 引入不可控风险；WebGPU 作为后续可选项单列。
- **粒子法并非一无是处**：若将来要"角色跳水激起水花"这类局部特效，可在高度场方案之上**局部叠加**少量粒子做点缀，但主体填充逻辑仍归高度场。
